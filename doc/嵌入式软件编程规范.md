# 嵌入式 C 编程规范

适用范围：MCU、RTOS、裸机、驱动、协议栈、BSP、应用层模块等 C 语言代码。

目标：代码清晰、稳定、易移植、易测试、易让 AI 理解和改造。

------

## 1. 总体原则

### 1.1 清晰优先

代码首先是给人看的，其次才是给机器执行的。

优先级：

```text
正确性 > 清晰性 > 可维护性 > 可测试性 > 性能优化
```

除非已经证明某处是性能瓶颈，否则不要为了所谓效率牺牲代码可读性。

### 1.2 简洁为美

避免大函数、大文件、大结构体、大宏。

重复代码出现 2 次时，应考虑提取函数；出现 3 次时，必须重构。

### 1.3 风格一致

新工程按本文规范执行。

维护旧工程时，优先保持原工程风格，不要在同一工程中混用多种命名、缩进、注释风格。

------

## 2. 文件与模块组织

### 2.1 `.h` 是接口，`.c` 是实现

头文件只放对外接口：

```c
void uart_init(void);
int uart_send(const uint8_t *data, uint16_t len);
```

不要在头文件中放：

```text
内部函数声明
内部变量定义
复杂实现逻辑
无关头文件
私有宏
```

### 2.2 每个 `.c` 文件优先配一个同名 `.h`

如果 `.c` 文件没有任何对外接口，应确认它是否真的需要独立存在。

内部函数应在 `.c` 文件内声明为 `static`。

```c
static void uart_rx_handler(void);
```

## 3. 头文件规范

### 3.1 必须有头文件保护

```c
#ifndef UART_H
#define UART_H

#ifdef __cplusplus
extern "C" {
#endif

/* public declarations */

#ifdef __cplusplus
}
#endif

#endif /* UART_H */
```

### 3.2 禁止头文件循环包含

禁止：

```text
a.h include b.h
b.h include a.h
```

如果两个模块互相依赖，说明设计可能有问题，应提取公共接口或使用前置声明。

### 3.3 只包含必要头文件

不要写万能头文件：

```c
#include "all.h"
#include "common.h"
```

每个 `.c/.h` 只包含自己真正需要的头文件。

### 3.4 头文件必须自包含

任意 `.h` 文件应能被单独 include 后正常编译。

也就是说，使用者不应该为了包含 `uart.h`，还必须提前包含其他隐藏依赖头文件。

## 4. 命名规范

### 4.1 推荐使用小写加下划线

```c
uart_init();
sensor_read_value();
motor_set_speed();
```

文件名也使用小写：

```text
uart_driver.c
sbus_protocol.c
device_manager.c
```

### 4.2 命名要表达含义

推荐：

```c
uint16_t rx_length;
uint8_t retry_count;
bool is_timeout;
```

避免：

```c
uint16_t len1;
uint8_t aa;
bool flag;
```

`flag`、`data`、`temp` 可以短期局部使用，但不要作为长期变量名滥用。

### 4.3 全局变量和静态变量加前缀

全局变量：

```c
uint8_t g_system_state;
```

文件内静态变量：

```c
static uint8_t s_uart_rx_buffer[128];
```

局部静态变量：

```c
static uint32_t s_call_count;
```

### 4.4 函数按动作命名

推荐：

```c
uart_send();
flash_erase_sector();
sensor_get_temperature();
motor_stop();
```

常用动词：

```text
init / deinit
start / stop
open / close
read / write
get / set
enable / disable
create / destroy
lock / unlock
send / receive
encode / decode
```

### 4.5 不使用拼音命名

禁止：

```c
dianji_start();
chuanganqi_read();
```

推荐：

```c
motor_start();
sensor_read();
```

## 5. 函数设计规范

### 5.1 一个函数只做一件事

函数名能准确描述函数行为。

如果函数名需要用 `and` 才能描述，通常说明函数应该拆分。

不推荐：

```c
void uart_init_and_start_dma(void);
```

推荐：

```c
void uart_init(void);
void uart_start_dma(void);
```

### 5.2 嵌套不超过 4 层

优先使用提前返回减少嵌套。

不推荐：

```c
if (ptr != NULL)
{
    if (len > 0)
    {
        if (state == READY)
        {
            do_work();
        }
    }
}
```

推荐：

```c
if (ptr == NULL)
{
    return ERR_NULL;
}

if (len == 0)
{
    return ERR_PARAM;
}

if (state != READY)
{
    return ERR_STATE;
}

do_work();
```

### 5.3 输入参数尽量使用 `const`

函数不会修改输入数据时，必须加 `const`。

```c
int protocol_parse(const uint8_t *frame, uint16_t len);
```

### 5.4 内部函数使用 `static`

只在当前 `.c` 文件内使用的函数，必须声明为 `static`。

```c
static uint8_t checksum_calc(const uint8_t *data, uint16_t len);
```

## 6. 变量与数据结构

### 6.1 变量必须初始化后再使用

禁止使用未初始化变量。

推荐变量定义时就初始化，且离使用位置越近越好。

```c
uint8_t retry_count = 0;
```

### 6.2 一个变量只表达一个含义

不要让同一个变量一会儿表示索引，一会儿表示返回值。

不推荐：

```c
uint8_t value;

value = index;
value = get_result();
```

推荐：

```c
uint8_t index;
int result;
```

### 6.3 少用全局变量

全局变量应尽量封装在模块内部，通过 API 访问。

不推荐外部直接访问：

```c
extern uint8_t g_uart_state;
```

推荐：

```c
uart_state_t uart_get_state(void);
```

### 6.4 通信结构体必须考虑字节序和对齐

协议帧、Flash 数据、网络数据不要直接依赖结构体内存布局。

谨慎使用：

```c
#pragma pack
__attribute__((packed))
```

推荐显式编解码：

```c
buffer[0] = (uint8_t)(value >> 8);
buffer[1] = (uint8_t)(value);
```

## 7. 宏、常量与枚举

### 7.1 禁止魔法数字

不推荐：

```c
if (timeout > 3000)
```

推荐：

```c
#define UART_TX_TIMEOUT_MS    3000U

if (timeout > UART_TX_TIMEOUT_MS)
```

### 7.2 宏表达式必须加完整括号

```c
#define ARRAY_SIZE(arr)    (sizeof(arr) / sizeof((arr)[0]))
#define MAX(a, b)          (((a) > (b)) ? (a) : (b))
```

### 7.3 多语句宏使用 `do { } while (0)`

```c
#define LED_ON()       \
    do                 \
    {                  \
        gpio_set(LED); \
    } while (0)
```

### 7.4 宏参数禁止带副作用

禁止：

```c
value = MAX(a++, b);
```

推荐：

```c
a++;
value = MAX(a, b);
```

### 7.5 优先使用 `const`、`enum`、`static inline`

常量优先：

```c
static const uint16_t UART_RX_BUFFER_SIZE = 128U;
```

枚举优先：

```c
typedef enum
{
    UART_STATE_IDLE = 0,
    UART_STATE_BUSY,
    UART_STATE_ERROR
} uart_state_t;
```

短小函数优先使用 `static inline`，不要滥用函数宏。

## 8. 指针、内存与字符串

### 8.1 指针使用前必须判断合法性

模块边界函数必须检查指针参数。

```c
if (data == NULL)
{
    return ERR_NULL;
}
```

### 8.2 明确缓冲区长度

涉及数组、字符串、协议帧时，函数参数必须带长度。

不推荐：

```c
int parse_frame(uint8_t *frame);
```

推荐：

```c
int parse_frame(const uint8_t *frame, uint16_t len);
```

### 8.3 避免动态内存

MCU 工程中默认不使用 `malloc/free`。

确实需要动态内存时，必须满足：

```text
有明确生命周期
有失败处理
无内存泄漏
无碎片风险
有统一封装接口
```

### 8.4 字符串操作必须限制长度

禁止无边界字符串操作：

```c
strcpy();
strcat();
sprintf();
```

推荐：

```c
snprintf();
strncpy();
```

但使用 `strncpy` 时仍需手动保证结尾 `'\0'`。

## 9. 中断、并发与实时性

### 9.1 ISR 中只做短小、确定的事情

中断服务函数中避免：

```text
长循环
阻塞等待
动态内存
复杂日志打印
耗时计算
```

推荐 ISR 只做：

```text
清中断标志
读取/保存关键数据
设置事件标志
通知任务处理
```

### 9.2 ISR 与主循环/任务共享变量必须使用 `volatile`

```c
static volatile bool s_uart_tx_done = false;
```

注意：`volatile` 只保证每次从内存读取，不保证原子性，也不保证线程安全。

### 9.3 多上下文共享数据必须保护

以下场景必须考虑临界区：

```text
ISR 与主循环共享
ISR 与 RTOS 任务共享
多个任务共享
DMA 与 CPU 共享
```

保护方式：

```text
关中断
临界区
互斥锁
信号量
原子操作
缓存一致性处理
```

### 9.4 状态机优先于复杂 if-else

协议、通信、设备流程建议使用状态机。

```c
switch (state)
{
case STATE_IDLE:
    break;

case STATE_RX:
    break;

case STATE_TX:
    break;

default:
    state = STATE_IDLE;
    break;
}
```

## 10. 错误处理与返回值

### 10.1 必须处理函数返回值

凡是可能失败的函数，调用后必须检查返回值。

```c
ret = flash_write(addr, data, len);
if (ret != ERR_OK)
{
    return ret;
}
```

### 10.2 统一错误码

### 10.3 错误路径要释放资源

有资源申请时，错误路径必须释放。

```c
ret = resource_lock();
if (ret != ERR_OK)
{
    return ret;
}

/* do something */

resource_unlock();
return ERR_OK;
```

复杂函数可使用统一出口，但不要滥用 `goto`。

## 11. 注释规范

### 11.1 注释解释“为什么”，不是重复“做了什么”

不推荐：

```c
i++; // i 加 1
```

推荐：

```c
/* 跳过帧头字节 */
index++;
```

### 11.2 文件头说明模块职责

```c
/**
 * @file uart.c
 * @brief UART 驱动模块，提供初始化、发送、接收接口。
 */
```

### 11.3 对外函数必须有注释

```c
/**
 * @brief 发送一段 UART 数据。
 * @param data 数据缓冲区指针，不能为 NULL。
 * @param len 数据长度，必须大于 0。
 * @return ERR_OK 表示成功，其它值表示失败。
 */
int uart_send(const uint8_t *data, uint16_t len);
```

### 11.4 特殊逻辑必须注释

以下情况必须写注释：

```text
协议特殊规则
硬件特殊时序
芯片 Errata 规避
故意 fall through
中断/并发保护原因
单位换算
魔法值无法消除时
```

## 12. 排版与格式

### 12.1 使用 4 空格缩进

不使用 Tab，或在工程中统一 Tab 宽度。

### 12.2 大括号独占一行

```c
if (condition)
{
    do_something();
}
else
{
    do_other();
}
```

### 12.3 每行只做一件事

不推荐：

```c
a = b + c; d = e + f;
```

推荐：

```c
a = b + c;
d = e + f;
```

### 12.4 表达式避免过度复杂

复杂条件应拆成有意义的布尔变量。

```c
bool is_valid_addr = (addr >= START_ADDR) && (addr < END_ADDR);
bool is_aligned = ((addr % 4U) == 0U);

if (is_valid_addr && is_aligned)
{
    flash_write(addr, data, len);
}
```

## 13. 可移植性

### 13.1 使用标准整数类型

优先使用：

```c
uint8_t
uint16_t
uint32_t
int8_t
int16_t
int32_t
size_t
bool
```

不要依赖 `int`、`long` 在不同平台上的长度。

### 13.2 硬件相关代码集中隔离

业务逻辑不要直接操作寄存器。

不推荐：

```c
GPIOA->ODR |= GPIO_PIN_1;
```

推荐封装：

```c
gpio_set(GPIO_LED);
```

### 13.3 平台适配层独立

推荐分层：

```text
app      应用逻辑
service  业务服务
driver   设备驱动抽象
bsp      芯片/板级适配
hal      厂商库或寄存器层
```

上层只依赖抽象接口，不直接依赖具体芯片。

## 14. 测试与自检

### 14.1 重要模块必须有基本测试

优先测试：

```text
协议解析
环形缓冲区
状态机
参数边界
错误路径
超时处理
数据转换
Flash 读写逻辑
```

### 14.2 边界条件必须覆盖

至少检查：

```text
NULL 指针
长度为 0
最大长度
越界长度
重复调用
超时
忙状态
异常状态
```

### 14.3 每次改代码后做自检

提交前检查：

```text
能否编译通过
是否有未使用变量
是否有未初始化变量
是否有返回值未处理
是否新增重复代码
是否破坏接口兼容
是否影响中断/并发安全
是否影响协议兼容
```

## 15. AI 代码改造专用规则

当把代码交给 AI 优化时，AI 必须遵守以下规则。

### 15.1 不允许随意改变外部接口

未经明确要求，不得修改：

```text
对外函数名
函数参数
返回值含义
结构体对外字段
协议格式
宏定义含义
文件名
模块边界
```

如果确实需要修改接口，必须先说明原因，并给出兼容方案。

### 15.2 优先做安全、清晰、低风险改造

AI 优先处理：

```text
函数拆分
变量命名优化
重复代码提取
返回值检查
NULL 指针检查
魔法数字替换
宏安全性修正
嵌套层级降低
注释补充
状态机整理
```

### 15.3 不做无依据的性能优化

AI 不得为了“可能更快”而改复杂代码。

只有满足以下条件才允许性能优化：

```text
明确性能瓶颈
不改变功能
不破坏可读性
能解释优化依据
```

### 15.4 保持嵌入式约束

AI 改代码时必须考虑：

```text
RAM/Flash 资源有限
中断上下文限制
RTOS 并发问题
volatile 变量语义
寄存器访问顺序
字节序
对齐
DMA 缓存一致性
无动态内存或少动态内存
```

### 15.5 输出改造结果时必须说明

AI 每次改代码后，应输出：

```text
1. 修改了什么
2. 为什么这么改
3. 是否改变外部接口
4. 是否有潜在风险
5. 建议测试点
```

## 16. 个人代码检查清单

写完代码后逐项检查：

```text
[ ] 文件职责是否单一？
[ ] 头文件是否只暴露必要接口？
[ ] 头文件是否有保护符？
[ ] 是否存在循环 include？
[ ] 函数是否只做一件事？
[ ] 函数是否过长？
[ ] 嵌套是否超过 4 层？
[ ] 参数是否超过 5 个？
[ ] 指针参数是否检查 NULL？
[ ] 数组/缓冲区是否检查长度？
[ ] 返回值是否全部处理？
[ ] 变量是否初始化？
[ ] 是否存在魔法数字？
[ ] 宏是否安全？
[ ] 全局变量是否可以减少？
[ ] ISR 中是否做了耗时操作？
[ ] 共享变量是否需要 volatile 或临界区？
[ ] 协议数据是否考虑字节序？
[ ] 是否有必要注释？
[ ] 是否能移植到其他 MCU？
[ ] 是否给出了测试点？
```

## 17. 最终判断标准

一段合格的嵌入式 C 代码应该满足：

```text
看得懂
改得动
测得了
移得走
跑得稳
出错能定位
让 AI 也能准确理解
```
